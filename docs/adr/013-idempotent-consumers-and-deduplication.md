# ADR-013: Idempotent consumers and deduplication

**Status:** Accepted

**Context:**
With at-least-once delivery semantics (ADR-004, ADR-009), a consumer may receive the same event more than once — due to producer retries, consumer rebalancing, or Kafka recovery. Processing a duplicate event must not corrupt the consumer's state (e.g. creating a duplicate payment, double-counting a fraud score).

**Decision:**
Every event consumer implements idempotency using one of two mechanisms:

- **Processed-events table** (recommended for services with a relational database): a table recording processed event IDs, checked within the same transaction as the state change. This gives atomic deduplication.
- **Redis with TTL** (for services where a database check is too expensive or where the service uses Redis already): the event ID is stored in Redis with a TTL long enough to cover the replay window.

The event envelope includes a globally unique event ID (UUID) generated by the producer, which serves as the idempotency key.

**Consequences:**
- Duplicate events are silently acknowledged without state corruption.
- Every consumer must check the idempotency key before processing — this is a mandatory implementation pattern, not optional.
- The processed-events table or Redis key space must be sized and cleaned to prevent unbounded growth.
- Idempotency does not solve ordering issues — out-of-order events are a separate concern addressed by partition keying (ADR-009).
